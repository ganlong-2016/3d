# 3D 到底是怎么回事？——用「造车」讲清楚 3D 原理

> 我们要做车机 3D 中控系统。这篇文章用「造车」的比喻，帮团队从零搞懂 3D 的所有核心概念。

---

## 一、3D 模型的本质：用三角形拼出一辆车

### 从钢板到车身

一辆真车的车身，是用很多块钢板焊接、冲压拼出来的。3D 模型也一样——**用大量三角形拼出物体的形状**。

为什么是三角形？因为三角形是最简单的「面」，三个点必然在同一个平面上，不会翘，GPU 处理起来最高效。

### 从点到面到整车

```
点（Vertex）         →    三角形（Triangle）      →    模型（Mesh）
焊接点                    一块三角形钢板              整个白车身

  •                       •───•                      ╱╲╱╲╱╲
                          │ ╱                        ╱╲╱╲╱╲╱╲
                          •                          ╲╱╲╱╲╱╲╱
                                                      ╲╱╲╱╲╱
```

- **顶点（Vertex）**：空间中的一个点，就像车身上的一个焊接点，有 x、y、z 三个坐标
- **三角形（Triangle）**：三个焊接点连起来，形成一块最小的钢板
- **网格（Mesh）**：所有三角形钢板拼在一起，就是整个 **白车身**

> Mesh 就是白车身——有了形状，但还没喷漆、没装内饰，光秃秃的。

### 面数：三角形越多，车身越精细

三角形的数量叫 **面数（Poly Count）**，决定模型精细程度：

- 车机中控里远处的小车：几千面 —— 看个轮廓就行
- 中控主界面的 3D 车模：几万面 —— 线条流畅、细节够看
- 营销展示级别的车模：几十万~上百万面 —— 车灯透镜、缝线都清晰

> 面数越多越精细，但 GPU 计算量越大。车机芯片性能有限，所以要在「好看」和「流畅」之间找平衡。

## 二、给白车身穿上外衣：纹理、材质、Shader

白车身有了形状，但看起来什么都不像。要让它变成你看到的那辆漂亮的车，需要 **喷漆、装内饰、做表面处理**。

这就对应 3D 中的三个东西：**纹理、材质、Shader**。

### 它们是什么关系？—— 一辆车的「外观工艺」

| 3D 概念 | 造车比喻 | 做什么 |
|---------|----------|--------|
| **Mesh（网格）** | 白车身 | 决定了车的形状——是 SUV 还是轿跑 |
| **纹理（Texture）** | 漆料、皮革样本、碳纤维纹样 | 就是一张张图片素材：车漆颜色、内饰皮革纹路、碳纤维花纹 |
| **材质（Material）** | 喷漆 / 内饰工艺方案 | 规定了用什么颜色、表面是高光还是哑光、是金属漆还是珠光漆 |
| **Shader（着色器）** | 喷漆车间的机器人 | 真正干活的——按照工艺方案，结合灯光环境，把每一寸车身喷涂出最终效果 |

它们的配合关系：

```
纹理（漆料/皮革纹样）──→ 喂给 ──→ 材质（工艺方案）──→ 交给 ──→ Shader（喷漆机器人）──→ 最终外观
                                                                    ↑
                                                              展厅灯光也会影响效果
```

### 纹理：各种「表面素材图」

纹理不只是颜色图，它有很多种，各司其职：

| 纹理类型 | 在车上对应什么 | 作用 |
|----------|---------------|------|
| 颜色贴图（Albedo） | 车漆颜色、内饰皮革颜色 | 物体的底色 |
| 法线贴图（Normal Map） | 皮革上的纹路、碳纤维的编织纹理 | 假装表面有凹凸（实际上模型表面是平的，但看起来有纹路） |
| 粗糙度贴图（Roughness） | 新车漆面（很光滑）vs 磨砂饰条（粗糙） | 控制哪里反光强、哪里不反光 |
| 金属度贴图（Metallic） | 车身钢板（金属）vs 保险杠塑料件（非金属） | 区分金属和非金属区域 |
| 环境遮蔽贴图（AO） | 车门缝隙、轮毂螺丝孔里自然更暗 | 让缝隙和角落更暗，增加真实感 |
| 自发光贴图（Emissive） | 尾灯亮起、仪表盘发光 | 控制哪些区域自己发光 |

#### 纹理怎么「贴」到模型上？—— UV 映射

模型是立体的，纹理是一张平面图片。怎么把平图贴到立体表面上？

> 想象你要给一辆车贴车衣。车衣出厂时是平的，你得知道哪块对应引擎盖、哪块对应车门、哪块对应后备箱，然后一块块贴上去，不能贴歪了。

这个「展平 + 对应」的过程就叫 **UV 映射**。模型上每个顶点记录一对 (u, v) 坐标，表示它对应纹理图上的哪个位置。

### 材质：一套完整的「表面工艺方案」

材质就是一份配置单：
- 车漆颜色：珍珠白
- 表面效果：金属漆（金属度 0.9）
- 光滑度：高光（粗糙度 0.1）
- 凹凸细节：用碳纤维法线贴图
- 缝隙暗部：用 AO 贴图

> 纹理是原材料（漆料、皮革卷），材质是工艺卡片（哪个零件用什么漆、喷几层、什么光泽度）。

现在主流的材质体系叫 **PBR（基于物理的渲染）**——模拟真实物理规律。你只需要调「金属度」和「粗糙度」，就能表达从轮胎橡胶到镀铬饰条的各种质感，在任何光照下都正确。

### Shader：真正干活的「喷漆机器人」

Shader 是一小段运行在 **GPU** 上的程序。它拿到材质的工艺方案和纹理素材，结合灯光信息，计算出每个像素的最终颜色。

> 同样的漆料和工艺方案，不同的 Shader（不同的喷漆机器人）可以喷出不同风格——写实风格（像真车照片）、卡通风格（像动画片里的车）、X 光风格（透视看到内部结构）。

Shader 主要有两种：

| 类型 | 造车比喻 | 干什么 |
|------|----------|--------|
| **顶点着色器（Vertex Shader）** | 确定每个焊接点在空间中的位置 | 把模型的每个顶点变换到正确的屏幕位置 |
| **片元着色器（Fragment Shader）** | 给白车身的每一寸表面喷漆上色 | 对每个像素计算最终颜色 |

## 三、场景：搭建一个完整的 3D 展厅

有了车模型和外观工艺，还只是准备好了一辆车。要让用户在中控屏上看到它，你还需要一个 **场景（Scene）**——一个完整的虚拟展厅。

### 场景里有什么？

> 场景就是一个 **汽车展厅**。车摆好了（模型），灯打好了（灯光），摄像机架好了（摄像机），背景幕布挂好了（天空盒），一切就绪，按下快门就能出片。

```
场景（Scene）—— 汽车展厅
├── 摄像机（Camera）        —— 看车的视角（可以绕车一圈看）
├── 灯光（Light）           —— 展厅的射灯、顶灯、氛围灯
├── 3D 车模（Mesh）         —— 展厅里摆着的那辆车
│   ├── 车身外壳
│   ├── 车轮 × 4
│   ├── 车灯
│   ├── 内饰（座椅、方向盘、中控台...）
│   └── 车窗玻璃
├── 地面 / 展台              —— 车停在什么上面
├── 天空盒（Skybox）         —— 展厅四周的环境（或户外背景）
└── 粒子 / 特效              —— 如尾灯光晕、轮胎烟雾等
```

### 摄像机（Camera）—— 看车的视角

摄像机决定了用户从什么位置、什么角度看这辆 3D 车。中控屏上看到的画面，就是摄像机「拍到」的画面。

| 模式 | 效果 | 车机场景中 |
|------|------|-----------|
| **透视投影（Perspective）** | 近大远小，有纵深感 | 3D 车模展示——绕车看一圈，车头近的时候大、车尾远的时候小 |
| **正交投影（Orthographic）** | 远近一样大 | 俯视泊车辅助画面、2D 仪表盘界面 |

摄像机有一个 **视锥体（Frustum）**——一个锥形的可见区域。只有在这个范围内的东西才会被画出来。

> 就像驾驶员的视野：正前方看得到，车后面看不到（除非有后视摄像头）。视锥体之外的东西直接跳过不画，省性能。

### 灯光（Light）—— 展厅里的灯

没有灯，展厅一片漆黑，再好的车也看不见。不同灯光营造不同效果：

| 灯光类型 | 车机场景类比 | 特点 |
|----------|-------------|------|
| **平行光（Directional Light）** | 模拟户外太阳光 | 光线平行，照亮整个场景 |
| **点光源（Point Light）** | 展厅天花板的筒灯 | 从一个点向四周发光，近处亮、远处暗 |
| **聚光灯（Spot Light）** | 展厅打在车身上的射灯 | 锥形光束，聚焦照亮某个区域 |
| **环境光（Ambient Light）** | 展厅的基础照明 | 均匀照亮所有角落，防止暗处全黑 |

灯光打到车身后，Shader 会算出：
- **车身上的明暗变化** —— 让车身有立体的光影弧线
- **车漆上的高光点** —— 让你觉得这个漆面很亮很光滑
- **车底的阴影** —— 让车有「停在地面上」的真实感

### 阴影（Shadow）

阴影不是自动有的，需要额外计算。最常用的是 **阴影贴图（Shadow Map）**：

> 假设灯光有一个「眼睛」。先从灯的视角看一遍场景，记录它能直接照到的最近表面。然后正常渲染时，每个像素都问：「灯能直接照到我吗？」照不到的就是阴影。

就像你在展厅里，射灯从上方打下来，车底和车身背光面自然就暗了。

### 天空盒（Skybox）—— 展厅四周的背景

> 天空盒就是展厅四周墙上的巨幅背景画。可以是户外风景、城市夜景、或者纯色渐变。你转头看哪个方向都有背景。

天空盒还有一个隐藏作用：**环境反射**。车漆表面反射出的「周围环境」，其实就是天空盒的图像。这就是为什么车漆看起来会倒映周围环境——它「照」的就是天空盒。

### 场景图（Scene Graph）—— 零件装配树

一辆车有几百个零件，怎么组织管理？用 **场景图**——一个树形结构，就像汽车零件的装配关系树：

```
整车（根节点）
├── 车身
│   ├── 引擎盖
│   ├── 左前门          ← 门打开时，门上的后视镜、门把手自动跟着动
│   │   ├── 后视镜         不需要单独给每个零件算位置
│   │   └── 门把手
│   ├── 右前门
│   └── 后备箱盖
├── 底盘
│   ├── 左前轮          ← 车身移动时，四个轮子自动跟着走
│   ├── 右前轮
│   ├── 左后轮
│   └── 右后轮
└── 内饰
    ├── 方向盘
    ├── 座椅
    └── 中控台
```

> 核心好处：移动父节点，子节点自动跟着动。开车门时只需要旋转「门」这个节点，门上的后视镜、门把手、车窗全都自动跟着转。

## 四、一辆 3D 车从数据到中控屏的完整旅程

这是最关键的部分——一辆 3D 车模的数据，经过了哪些步骤，最终出现在车机的中控屏上？

### 用「造车流水线」来比喻

```
第一步：备料和排产计划       →  CPU 准备 3D 数据
第二步：零件定位，装到车身上  →  顶点变换（把模型放到世界中正确位置）
第三步：从你的角度看这辆车    →  摄像机变换（从用户视角看）
第四步：拍一张照片           →  投影（3D → 2D，立体变平面）
第五步：照片打印成像素点      →  光栅化（三角形 → 像素格子）
第六步：给每个像素涂上颜色    →  着色（Shader 喷漆上色）
第七步：照片送到屏幕展示      →  显示到中控屏上
```

### 详细走一遍

#### 第一步：CPU 备料排产 —— 「总工程师安排工作」

CPU（总工程师）做准备：

- 场景里有哪些模型？车身、车轮、地面、背景……
- 摄像机在哪？用户现在看的是车的哪个角度？
- 灯光怎么打？几盏灯、什么颜色、什么方向？
- **视野之外的东西直接跳过不画**（车的背面看不到就不画、屏幕外的地面不画）

跳过看不见的东西叫 **剔除（Culling）**——就像排产时跳过暂时不需要的零件。

准备好后，CPU 给 GPU 下达指令：「用这套车漆工艺画这批零件的三角形」。这条指令叫 **Draw Call**。

> Draw Call 就像总工程师给流水线下的 **工单**。工单越多，总工越忙。如果能把几个小零件合并成一张工单（**批处理 Batching**），效率就高得多。

#### 第二步 & 第三步：顶点变换 —— 「零件装配定位」

每个三角形的顶点要经过一连串坐标变换：

1. **模型空间 → 世界空间**：零件从仓库（自己的坐标系）搬到总装车间的正确工位（世界坐标）。比如左前轮装到车身左前方的位置。
2. **世界空间 → 摄像机空间**：用户坐进驾驶舱（或绕到车前方），所有东西从用户的视角重新排列——什么在左边、什么在右边、什么近什么远。
3. **摄像机空间 → 屏幕**：用户「拍一张照片」——三维世界被压成二维平面图像（近大远小的透视效果）。

这三步用三个矩阵完成，合称 **MVP 变换**（Model-View-Projection）。

#### 第四步：光栅化 —— 「把照片变成像素点阵」

现在每个三角形都有了屏幕上的 2D 坐标。这一步要做的是：**找出每个三角形覆盖了屏幕上的哪些像素**。

> 就像你在方格纸上画一个三角形，然后数它盖住了哪些格子。

#### 第五步：着色 —— 「喷漆机器人上色」

对每一个像素，**Fragment Shader（喷漆机器人）** 会：

1. 查一下这个像素对应纹理上的哪个位置，取出底色（珍珠白车漆）
2. 看一下法线贴图，算出这个点表面的朝向（皮革纹路的凹凸方向）
3. 根据灯光方向，算出这里有多亮、高光有多强（车漆上的高光弧线）
4. 结合粗糙度、金属度，算出最终颜色（是亮闪闪的金属漆还是哑光塑料）

> 这就是为什么 Shader 叫「着色器」——它真的在一个像素一个像素地「喷漆」。

#### 第六步：深度判断与混合 —— 「谁挡住了谁」

如果两个零件在屏幕上重叠了（比如车门挡住了座椅），怎么决定显示谁？靠 **深度缓冲（Z-Buffer）**：每个像素记住自己离摄像机多远，近的覆盖远的。

> 就像从车外看，你只看到车门，看不到里面的座椅——车门离你近，挡住了后面的一切。

如果是半透明的东西（比如车窗玻璃），就不能简单用前面覆盖后面，要做 **混合（Blending）**——把玻璃的颜色和后面的内饰颜色按透明度混在一起。这就是车窗能「透过去看到内饰」的原理。

#### 第七步：送到中控屏 —— 「下线交付」

所有像素的颜色都算好了，存在 **帧缓冲（Frame Buffer）** 里。中控屏的显示芯片读取这块内存，画面就出来了。

一秒做 60 次 = 60fps。用 **双缓冲**：一帧在屏幕上显示，下一帧在后台偷偷画着，画完立刻换上去，避免画面撕裂。

## 五、让车动起来：动画与物理

到目前为止，我们的 3D 车是静止的展示品。但车机中控的 3D 界面里，车门要能开关、车轮要能转、氛围灯要能亮灭。

### 骨骼动画 —— 给车装上「铰链和转轴」

> 就像真车有车门铰链、方向盘转轴、座椅滑轨一样。3D 模型内部也藏着一副 **骨骼（Skeleton）**——一组可以活动的转轴/关节。

流程：
1. **绑骨骼**：在车模型内部设置骨骼——车门有铰链骨骼、车轮有旋转骨骼、引擎盖有翻转骨骼
2. **蒙皮（Skinning）**：把模型表面的顶点「绑」到对应骨骼上——车门表面的所有顶点绑到车门铰链骨骼上
3. **做动画**：只需要记录骨骼的运动，模型表面自动跟着变形——录制铰链从 0° 旋转到 60°，车门就自动开了

> 车门开关、天窗开合、后视镜折叠、方向盘转动……都可以用同一套骨骼系统来做。同一个车模，换不同的动画数据就行。

#### 关键帧（Keyframe）

动画不需要录制每一帧的状态，只需要在关键时刻打一个标记：
- 第 0 帧：车门关着（0°）
- 第 30 帧：车门打开到最大（60°）

中间的帧由计算机自动平滑过渡（**插值**），车门就平滑地「打开」了。

#### 动画混合（Blending）

如果同时要开车门和折叠后视镜，两个动画可以 **混合** 同时播放，互不干扰。

### 粒子系统（Particle System）—— 尾气、灯光、雨雪

有些效果没法用普通模型做：尾气烟雾、车灯的光晕、雨天的雨滴、轮胎打滑的烟尘……

这些靠 **粒子系统**：发射大量的小面片（粒子），每个粒子有位置、速度、大小、颜色、寿命。

> 就像一台迷你烟雾机。调节喷射速度、方向、烟雾浓度、消散时间，就能模拟出排气管的尾气、冬天的哈气、或者轮胎的扬尘。

在车机中控场景中常见的粒子效果：
- 尾灯拖尾光效
- 车底氛围灯的光晕
- 下雨/下雪天气效果
- 车辆启动时排气管的烟雾

### 物理引擎 —— 让 3D 世界有物理规则

3D 世界默认没有物理规则——车轮不会转，车不会掉到地上，碰到墙也穿过去。

**物理引擎** 模拟真实物理：

| 概念 | 车机场景类比 |
|------|-------------|
| **刚体（Rigid Body）** | 给车加上「重量」，它就会受重力影响、被碰撞推动 |
| **碰撞器（Collider）** | 给车套一个简化的「隐形外壳」用来检测碰撞。不用精确到每个三角形，用几个盒子近似就行 |
| **碰撞检测** | 检测两个物体是否碰在一起——比如车轮是否接触地面 |
| **射线检测（Raycasting）** | 从一个点发射一条「隐形激光」看碰到什么——**用户在中控屏上点击 3D 车模时，就是从摄像机发一条射线，看它碰到了车的哪个部位**（车门？车轮？引擎盖？） |

> **碰撞器比喻**：做碰撞测试时，不会把真车的每一个曲面都拿来算，而是套几个简化的盒子——一个大盒子罩住车身，四个小圆柱套住车轮。简单形状判断起来极快。

## 六、画面更好看：后处理与优化

渲染完一帧后，还可以对整张画面做「修图」让效果更好，也有一些手段让画面更流畅。

### 后处理（Post-Processing）—— 给「照片」修图

渲染完成的画面就像刚拍完的原片，后处理就是修图：

| 效果 | 车机场景中 | 比喻 |
|------|-----------|------|
| **泛光（Bloom）** | 车灯亮起时向外溢出的光晕 | 晚上看到路灯周围的那圈光 |
| **景深（Depth of Field）** | 聚焦到车身某个细节，周围虚化 | 汽车广告大片的虚化效果 |
| **运动模糊（Motion Blur）** | 车辆行驶时的速度感拖影 | 赛车画面中的速度感 |
| **色彩校正（Color Grading）** | 白天模式暖色调、夜间模式冷色调 | 像手机滤镜调色 |
| **环境光遮蔽（SSAO）** | 车底、轮拱内侧、缝隙处更暗 | 现实中这些地方确实更暗 |

### 抗锯齿（Anti-Aliasing）—— 让车身线条更平滑

因为屏幕是像素方格组成的，车身的弧线边缘会出现「锯齿」（阶梯状）。抗锯齿就是把锯齿变平滑。

> 就像车身钣金边缘要打磨光滑。如果不处理，弧形线条在屏幕上看起来像楼梯台阶。

### LOD（Level of Detail）—— 远处的零件不用那么精细

3D 中控的鸟瞰图里，车在远处时屏幕上只有很小一块，用高精度模型完全浪费。

**LOD** 就是准备同一辆车的多个版本：
- **近看**：高精度，连车灯内部透镜、缝线都有
- **中距离**：中等精度，大的线条还在，细节省了
- **远看**：简化版，几百面的轮廓就够了

> 就像你在 4S 店近距离看一辆车能看到缝线和漆面纹理，但在马路对面看一辆车只能看到个大致轮廓。没必要对远处的车渲染缝线细节。

### 实例化（Instancing）—— 停车场里的同款车

如果场景里有个停车场，停了 50 辆同款车。普通做法是下 50 次工单让 GPU 画 50 次，CPU 累死。

**Instancing**：只告诉 GPU 一次「车长这样」，附上 50 个位置，GPU 自己批量复制。

> 就像汽车工厂：不是每辆车都从头造，而是同一条流水线用同一个模具，只改颜色和配置，批量下线。

### 烘焙（Baking）—— 提前把效果「烤」好

有些效果实时计算太贵，可以提前算好存成图片，运行时直接用：

| 烘焙类型 | 车机场景类比 |
|----------|-------------|
| **光照烘焙** | 展厅的灯光永远不变，就提前把光影效果算好「烤」成一张贴图。运行时直接贴上去，不用每帧重新算光照 |
| **AO 烘焙** | 车底、轮拱缝隙的暗部永远暗，提前算好存成贴图 |
| **法线贴图烘焙** | 用一个几百万面的超精细车模，把表面细节「烤」成一张法线贴图，贴到只有几万面的简化车模上——看起来一样精细，但面数少几十倍 |

> 就像新车出厂前，把保护膜贴好、漆面抛好光——这些工序在工厂里（离线）完成，交付后（运行时）就不用再做了。

核心取舍：烘焙用 **存储空间换计算速度**。好处是运行时几乎零开销；代价是「烤死了」就不能变——灯位置固定、车位置不能动。适合 **不会变的静态场景**。

## 七、坐标与空间：3D 世界的「定位系统」

每个 3D 物体需要一个位置。3D 用三个轴描述位置：

- **X 轴**：左右
- **Y 轴**：上下
- **Z 轴**：前后

任何一个点用 (x, y, z) 三个数定位。

### 几个重要的「坐标空间」

一个顶点从模型数据到最终屏幕，要在几个不同的坐标空间之间转换：

| 空间 | 车机比喻 |
|------|----------|
| **模型空间（Model Space）** | 车轮的坐标：原点在轮毂中心，所有点相对于轮毂说话 |
| **世界空间（World Space）** | 车轮装到车上，车停在展厅里。现在用展厅的坐标说话——车轮在展厅的 (2, 0, 3) 位置 |
| **摄像机空间（View Space）** | 用户站在展厅某个位置看车。从用户的角度重新定义「什么在左边、什么在前面」 |
| **屏幕空间（Screen Space）** | 最终显示在中控屏上的像素位置 |

> 就像描述一个车轮的位置：可以说「在轮毂正上方 30cm 处」（模型空间），也可以说「在展厅东北角那辆车的左前方」（世界空间），也可以说「在你正前方 2 米、偏左 1 米」（摄像机空间），最后变成「屏幕上第 300 行第 500 列的像素」（屏幕空间）。说的是同一个点，参照系不同。

## 八、与 3D 车模互动

车机中控的 3D 界面不只是「看」，用户还要能 **点击、滑动、操作**。比如点击车门就打开车门，点击车灯就开灯。

### 射线拾取（Raycasting / Picking）

中控屏是一块 2D 触摸屏，但上面显示的是 3D 场景。用户手指点下去，怎么知道点的是车的哪个部位？

> 想象用户的手指就是一根激光笔。从摄像机位置穿过手指点击的那个像素，射出一条「隐形光线」到 3D 世界中。光线碰到的第一个东西，就是用户点中的——可能是车门、可能是引擎盖、可能是轮毂。

这就是 **射线拾取**。在车机 3D 中控中：
- 点击车门 → 播放开门动画
- 点击车灯 → 切换灯光开关
- 点击座椅 → 进入座椅调节界面
- 点击轮毂 → 显示胎压信息

### 碰撞与触发区域

除了点击，还可以设置 **触发区域**：
- 3D 车模旋转到某个角度 → 自动高亮显示该区域的功能
- 用户手指滑过车身 → 变换车漆颜色预览

## 九、常见问题

### Q：GPU 和 CPU 各干什么？

| | CPU | GPU |
|--|-----|-----|
| **擅长** | 复杂逻辑判断，一件一件处理 | 简单但大量的重复计算，同时干几千件 |
| **车机比喻** | 总工程师：看图纸、排产计划、协调各部门 | 流水线上几千个工人：每人只做简单的拧螺丝，但同时干 |
| **在 3D 中** | 准备数据、处理交互逻辑、给 GPU 下工单 | 同时处理几百万个顶点和像素的计算 |

> 设计一辆车的方案，总工程师（CPU）快。但同时给一千辆车拧螺丝，流水线工人（GPU）碾压总工。渲染就是后者——每帧几百万个像素，每个像素做类似的计算，GPU 就是为此而生。

### Q：为什么有时候画面会卡？

1. **CPU 瓶颈**：总工下工单太慢，流水线（GPU）在等。常见原因是 Draw Call 太多。
2. **GPU 瓶颈**：三角形太多、Shader 太复杂、分辨率太高，流水线忙不过来。

> 总工安排得太慢，工人就闲着（CPU 瓶颈）。或者零件太精细、工艺太复杂，工人干不完（GPU 瓶颈）。车机芯片比 PC 弱很多，这两个瓶颈都要特别注意。

### Q：什么是实时渲染和离线渲染？

| | 实时渲染 | 离线渲染 |
|--|----------|----------|
| **速度** | 1 秒画 30~60 帧 | 1 帧算几秒到几小时 |
| **画质** | 够好就行，流畅优先 | 极致真实 |
| **用在哪** | 车机中控 3D 界面、游戏 | 汽车广告 CG、展厅效果图 |
| **车机比喻** | 展厅里实时转动的展车 | 汽车宣传海报的精修大片 |

### Q：什么是帧率（FPS）？

FPS = Frames Per Second，每秒画多少帧。

- **30 fps**：每帧 33 毫秒，基本流畅
- **60 fps**：每帧 16 毫秒，丝滑顺畅

> 车机中控至少要 30fps 才不会让用户觉得卡。所有优化手段（LOD、剔除、烘焙、Instancing、减少 Draw Call）都是为了让每帧的计算量控制在这个时间预算内。

### Q：3D 文件格式怎么区分？

| 格式 | 比喻 |
|------|------|
| **.glTF / .glb** | 标准提车配置——Web/车机端首选，体积小加载快 |
| **.fbx** | 完整工程档案——设计师交付用，信息最全 |
| **.obj** | 老款合格证——格式古老简单，但功能有限 |

### 一句话记住

> **3D 中控就是：用三角形焊出白车身（Mesh），用车漆和皮革纹理贴上外观（Texture），用工艺方案调出质感（Material），让喷漆机器人按灯光环境逐像素上色（Shader），加上铰链让车门能开合（动画），加上碰撞检测让用户能点选（交互），展厅灯光一打，摄像机一拍，一秒拍几十张——就是你在中控屏上看到的 3D 画面。**
